# Техревью и план миграции: как сделать `main` похожим на `sandbox`

Дата: 2026‑04‑21  
Репозиторий: `djalben/xplr-core`  
Цель: довести ветку `main` до архитектурных принципов и инженерных требований, реализованных в `sandbox`.

## Контекст и позиция

Я смотрю на это как senior Go engineer с опытом архитектуры микросервисов: важны **границы**, **явные зависимости**, **тестируемость**, **управляемая эволюция**, а не “красивое дерево папок”.

`sandbox` уже показывает правильное направление: Clean Architecture “по смыслу” (слои + порты + DI), автотесты, линтеры, единая инфраструктура логирования/ошибок.

Задача — сделать `main` таким же “скелетом”, сохранив функциональность.

---

## 1) Главные замечания к `main` (почему он не как `sandbox`)

### 1.1. Глобальные синглтоны и скрытые зависимости

В `main` есть pattern вида:
- `handler.GlobalDB = DB`
- `repository.GlobalDB = DB`

Проблемы:
- зависимости неявны (код можно вызвать без инициализации и получить runtime‑ошибки),
- тесты становятся дорогими/хрупкими (сложно подменять БД/репозитории),
- архитектура “плывёт” — любые пакеты начинают тянуть друг друга через globals.

**Цель как в `sandbox`**: зависимости передаются через конструкторы, границы фиксируются интерфейсами в `ports/`.

### 1.2. `backend/domain` в `main` не является доменом

Признак: `domain/models.go` импортирует `database/sql` (и в целом содержит DTO/контракты “для API”).

Это означает “протечку инфраструктуры” в домен. В итоге домен нельзя переносить, тестировать, развивать независимо.

**Цель как в `sandbox`**: домен — сущности/инварианты/ошибки/типы, без знания про HTTP/SQL/внешние библиотеки (кроме согласованных общих, если вы так решили — например `wrapper`).

### 1.3. `main.go` делает слишком много (god-init)

`main` сейчас совмещает:
- env загрузку
- подключение к БД + пинги
- проверку схемы/таблиц
- запуск воркеров
- конфигурацию SMTP/TG
- сборку router и регистрацию всех маршрутов

Это ломает “composition root” и усложняет изменения.

**Цель как в `sandbox`**: entrypoint минимален — config → logger → container → server.Start → graceful shutdown.

### 1.4. Нет жёсткой границы между слоями

В `main` usecases/handlers/services легко начинают импортировать друг друга напрямую. Отсутствует строгий договор: “usecase зависит от портов, а не от postgres/http”.

**Цель как в `sandbox`**: слой `application` зависит только от `domain` + `ports`. Инфраструктура — адаптеры, transport — thin handlers.

## 2) “Требования sandbox‑style”: чек‑лист, которому должен следовать тиммейт

Ниже — требования в формате “делай так всегда”, чтобы `main` стал похож на `sandbox`.

### 2.1. Структура слоёв (обязательная)

- **`internal/domain/`**
  - Сущности, value objects, инварианты, enum/типы, доменные ошибки.
  - Не зависит от `transport/`, `infrastructure/`, конкретной БД, роутера, SMTP, Telegram и т.п.
- **`internal/ports/`**
  - Интерфейсы “на границе” приложения: `UserRepository`, `Mailer`, `CardProvider` и т.д.
  - Никакой бизнес‑логики.
- **`internal/application/<feature>/`**
  - UseCase, реализующий бизнес‑правила.
  - Зависимости только через `ports` + типы `domain`.
  - Возвращает доменные ошибки (через `wrapper`, если принято).
- **`internal/infrastructure/`**
  - Реализации портов: postgres repositories, SMTP mailer, logger handler, внешние клиенты.
  - Можно зависеть от `sqlx`, `net/smtp`, внешних SDK и т.п.
- **`internal/transport/http/`**
  - Роутинг/handlers/middleware.
  - Handler не содержит бизнес‑логики: максимум “валидируй вход → вызови UC → маппинг ошибок → JSON”.

### 2.2. Зависимости (Dependency Rule)

- `domain` не импортирует `transport`/`infrastructure`.
- `application` не импортирует postgres/http/SMTP напрямую.
- Все внешние эффекты идут через порты (`ports.*`).

### 2.3. DI и composition root

- Вся сборка зависимостей делается в контейнере (`internal/app/container.go`) и entrypoint.
- Никаких глобальных `GlobalDB`.
- Конструкторы принимают зависимости явно.

### 2.4. Ошибки и логирование

- Ошибки оборачиваются через `wrapper` (у вас это стандарт).
- `domain` хранит sentinel errors; usecases возвращают `errors.Is`‑совместимые ошибки.
- В transport слое есть единый mapper `error -> HTTP code`.

### 2.5. Тесты и качество

- Unit тесты usecase’ов с моками портов.
- Handler тесты: проверка статусов/JSON/маппинга ошибок.
- `make test`, `make lint` зелёные.
- Нейминг без опечаток (`commission`, `env.example` и т.п.).

## 3) Рекомендации: как переделать `main` в `sandbox`‑style (пошаговый план)

### Шаг 0. Зафиксировать целевую структуру в README

- Обновить корневой `README.md`: описать архитектуру backend так же, как реально в `sandbox` (слои `internal/*`).
- Добавить правило “куда класть код” (domain/application/ports/infrastructure/transport).

### Шаг 1. Выделить новый backend‑каркас внутри `main`

В ветке `main` создать `backend/internal/...` и начать перенос “по вертикальным срезам”:
- Auth (регистрация/логин/email/2FA)
- User profile
- Wallet
- Cards
- Transactions
- Tickets

Важно: переносить **фичами**, а не “папками”, чтобы каждый шаг давал рабочий код.

### Шаг 2. Убрать globals и сделать явные зависимости

- Удалить `handler.GlobalDB`, `repository.GlobalDB`.
- Postgres доступ реализовать как адаптер `internal/infrastructure/persistence/postgres`, который удовлетворяет `ports.*Repository`.
- Все usecases переподключить через `container`.

### Шаг 3. Переписать transport слой

- Router/handlers переехать в `internal/transport/http`.
- `main.go` оставить только композицию:
  - parse config
  - init logger
  - build container
  - build server/router
  - start + graceful shutdown

### Шаг 4. Конфиг и окружение

- Перенести config в `internal/config` (через `envparse` как у вас принято).
- Добавить валидацию обязательных полей на старте.
- Для serverless (Vercel) — отдельный адаптер `vercel/http_handler.go`, который создаёт handler через тот же контейнер.

### Шаг 5. Выстроить тесты как в `sandbox`

- На каждый usecase: базовые happy‑path + ошибки (`errors.Is`).
- На handlers: хотя бы smoke (200/400/401/500) + контракты JSON.
- Интеграционные тесты репозиториев (SUT) — по готовности.

---

## 4) Критерии “готово” (Definition of Done)

Считать `main` “как sandbox” можно только когда:
- Нет глобальных зависимостей для DB/репозиториев/сервисов.
- Usecases зависят от `ports`, а не от конкретной реализации.
- `domain` не содержит HTTP DTO/SQL типов.
- `make test` и `make lint` зелёные.
- Entry point минимальный, как в `sandbox`.

---

## 5) Практический совет по стратегии миграции

Самый безопасный путь — “Strangler Fig”:
- новый `internal/*` появляется параллельно старым пакетам,
- роуты постепенно переводятся на новые handlers/usecases,
- старый код удаляется по мере исчерпания ссылок,
- на каждом шаге тесты/линтер зелёные.
