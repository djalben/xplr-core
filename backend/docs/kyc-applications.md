# KYC-заявки в XPLR — как это работает

В системе есть **два связанных уровня**:

1. **`users.kyc_status`** — текущий статус пользователя в продукте: `PENDING`, `APPROVED`, `REJECTED` (видно, например, в `GET /api/v1/user/me` как `kyc_status`).
2. **Таблица `kyc_applications`** — отдельные **заявки** с историей: кто подал, когда, что прислал в payload, что решил админ.

Заявка — это строка с полями: `user_id`, `status` заявки (`PENDING` / `APPROVED` / `REJECTED`), `payload_json` (произвольный JSON текстом), опционально `admin_comment`, `reviewed_by`, `reviewed_at`, `created_at`.

---

## Сценарий по шагам

### 1. Регистрация

У нового пользователя в `users` уже стоит **`kyc_status = PENDING`** (так задано в домене при создании пользователя). Отдельной заявки в `kyc_applications` ещё нет, пока пользователь сам ничего не отправит.

### 2. Пользователь подаёт заявку

- **Запрос:** `POST /api/v1/user/kyc/application` с JWT пользователя.
- **Тело:** например `{ "payload": { "fullName": "...", "documentId": "..." } }` — содержимое `payload` сериализуется в JSON и кладётся в `payload_json` заявки.
- **Правила:**
  - Если у этого пользователя **уже есть заявка в статусе `PENDING`**, вторую подать нельзя → ошибка вида *«kyc application already pending»*.
  - Если последняя заявка была отклонена или одобрена, **новая заявка создаётся заново** (ограничение только на одну *ожидающую* заявку).
- В БД создаётся запись в `kyc_applications` со статусом **`PENDING`**. Поле **`users.kyc_status` при подаче заявки не меняется** — оно обновится только после решения админа.

### 3. Админ смотрит очередь

- **Запрос:** `GET /api/admin/kyc-applications` (с JWT **админа**).
- Возвращается список заявок со статусом **`PENDING`** (до 200 штук): `id`, `user_id`, `payload`, `created_at` и т.д.

### 4. Админ принимает решение

- **Запрос:** `PUT /api/admin/kyc-applications/{id}/decision`
- **Тело:** `{ "approve": true/false, "comment": "опционально" }`
- **Правила:**
  - Работает только если заявка сейчас в **`PENDING`**. Иначе — ошибка *«application is not pending»*.
- **Что делает бэкенд** (последовательно: обновление пользователя, затем заявки):
  1. Обновляет **заявку**: `APPROVED` или `REJECTED`, пишет `reviewed_by` (id админа), `reviewed_at`, при необходимости `admin_comment`.
  2. Обновляет **`users.kyc_status`** у автора заявки: при одобрении → **`APPROVED`**, при отказе → **`REJECTED`**.

### 5. Пользователь после решения

- В **`GET /api/v1/user/me`** он видит актуальный **`kyc_status`** (`APPROVED` / `REJECTED` / пока без решения — как было до админа, обычно `PENDING`).
- При отказе может снова вызвать `POST .../kyc/application` и подать **новую** заявку (пока нет другой `PENDING`).

---

## Важные нюансы

| Тема | Как сейчас в коде |
|------|-------------------|
| Формат данных KYC | Любой JSON в `payload`; схема на стороне клиента/документации, бэкенд не валидирует поля. |
| Одна очередь | Админ видит только **ожидающие** заявки; одобренные/отклонённые в этом списке не показываются. |
| Повторная подача после одобрения | Технически возможна (нет активной `PENDING`), если фронт это разрешит — появится вторая заявка; продуктово это стоит явно зафиксировать. |
| Связь с картами/лимитами | В этом use case **нет** проверок «можно ли выпускать карту без KYC» — только хранение заявок и выставление `users.kyc_status`. |

---

## Кратко

Пользователь один раз (за раз) шлёт заявку с JSON-payload → админ из списка `PENDING` одобряет или отклоняет → обновляются и строка заявки, и `kyc_status` пользователя.
